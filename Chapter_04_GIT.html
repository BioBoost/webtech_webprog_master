What is Git

Git

  • English slang for a stupid or unpleasant person.
  • Initially designed and developed by Linus Torvalds for Linux kernel development in 2005.
  • Revision Control
    • Management of changes in documents
    • Trunks, Branches and Tags

Git

  • Source Code Management
    • Management of changes in documents
    • Trunks, Branches and Tags

Git

  • Distributed
    • Every Git working directory is a full-fledged repository with complete history and full version tracking capabilities
    • Not dependent on network access or a central server.
  • Fast

Installing and configuring Git

Installing Git

  • Download Git binary from http://git-scm.com
  • On windows choose "Run Git and include Linux tools from the Windows command Prompt" during installation

Configuring Git

  • Configuring your identity in Git

$ git config --global user.name = "Lorem Ipsum"
$ git config --global user.email = "lorem@ipsum.be"
$ git config --global core.editor "'C:\Program Files\Sublime Text 3\sublime_text.exe'"

This can also be done for a single repository by omitting --global.

  • Enable colors

$ git config --global color.ui true

Staging and commiting files

Before we are able to track files we need to add them to the repository. This is a two step process.

  1. Staging
    • Preparing for a commit by adding them to the staging area (the index)

D:\demo\git [master +1 ~0 -0 !]> git add file.txt

Staging and commiting files

  1. Commiting
    • Actually saving the changes to the repository

D:\demo\git [master +1 ~0 -0]> git commit -m "Added new file.txt"
[master (root-commit) 640e020] Added new file.txt
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 file.txt

Staging and commiting files

  • The current version of file.txt is now saved in the history of the repository.
  • Every commit needs a message
    • Short explanation of the changes
  • The commit is identified as 640e020
    • You can always come back to this version

Example


D:\demo\git [master]> touch hello.txt
D:\demo\git [master +1 ~0 -0 !]> echo "Lorem ipsum" >> .\file.txt

Git File Status Life Cycle

What to commit ?

  • Commit only per set of change
    • 1 commit per feature/bug/update/refactoring
    • Use $ git add –p and decide per code line (patch) if you want to add it to the staging area
  • The commit is identified as 640e020
    • You can always come back to this version

Adding and resetting

  • Add all files in current directory at once
    • $ git add .
  • Add all files with filename filter
    • $ git add *.cpp
  • Add a subfolder
    • $ git add subfolder/
  • Removing tracked file from the staging area
    • $ git reset file.txt
  • Reset your working copy to the latest committed state (undo all changes to tracked files)
    • $ git reset --hard

Adding and resetting

  • Reset your working copy of a single tracked file
    • $ git checkout file.txt

Removing files

  • Remove a file from the repository
    • $ git rm file.txt
    • Removing the file from the working directory won't remove the file in te repository. You must use the git rm command.

Branching

Branching

  • The entire history can be visualized as a timeline
  • A timeline is also called a branch
    • Multiple branches can exist in one project
    • By default git adds the history to the master branch

Branching

  • You can add your own branch with the git branch command

$ git branch testing

When to branch ?

  • Do every new experiment/feature/bug/... in a new branch
    • This way you will never break your working code base
    • If you ruin your project, just checkout a working revision, branch or tag.

The HEAD

  • The HEAD is a pointer that points to the current checked out commit
  • The HEAD is a symbolic reference to the branch you're currently on.

Moving the HEAD

  • With the checkout command we can move the HEAD.

$ git checkout testing

Committing on the testing branch

  • Adds a new revision (C5) to the history
  • 'testing' and HEAD are pointing to the new revision (C5)
  • Changes are added to the active branch
    • Only 1 branch can be active at any given time

Back to the master

  • Changing back to the master branch
    • The current working copy is moved back to the last state of master

$ git checkout master

Committing on the master branch

  • Make a new commit on the master branch
    • Testing branch is unaffected
    • Branches exist next to each other
    • Each branch has it's own history

Merging

Before Merging

  • Combine the changes from another branch in your current branch
  • git merge command
  • You always merge into your current checked out branch

Before Merging


$ git checkout iss53 
Switched to branch "iss53" 
$ vim index.html 
$ git commit -a -m 'finished the new footer [issue 53]' 
[iss53]: created ad82d7a: "finished the new footer [issue 53]“
 1 files changed, 1 insertions(+), 0 deletions(-)

Merging

  • Merge a branch into your current branch

$ git checkout master 
$ git merge iss53 
Merge made by recursive. 
README | 1 + 1 files changed, 
 1 insertions(+), 0 deletions(-)

After Merging

  • You can delete the branch after merging
  • Git will warn you if you didn't merge the branch first

$ git branch -d iss53

Some more merging commands

  • Listing all branches
    • $ git branch

Merging conflicts

  • When merging, problems can occur
    • e.g. the same line of code has been edited
    • When a merge conflict occurs, the merge will fail
    • Manual selection of which version to keep is needed
  • Solution:
    • Fix the problem and commit your changes !

$ git merge iss53 
Auto-merging index.html 
CONFLICT (content): Merge conflict in index.html 
Automatic merge failed; fix conflicts and then commit the result.

Merging conflicts

  • Git status will show merging conflicts

[master*]$ git status 
index.html: needs merge
# On branch master 
# Changes not staged for commit: 
# (use "git add <file>..." to update what will be committed) 
# (use "git checkout -- <file>..." to discard changes in working directory) 
# 
# unmerged: index.html 
#

Merging conflicts

  • If a merge fails, the file will contain both versions
  • You can manually edit the file for the correct version

<<<<<<< HEAD:index.html 
<div id="footer">contact : email.support@github.com</div> 
======= 
<div id="footer"> 
please contact us at support@github.com 
</div> 
>>>>>>> iss53:index.html

Merging conflicts

  • Checkout the correct version
    • Choose version from HEAD

$ git checkout --ours conflictedfile
    • Choose version from otherbranch

$ git checkout --theirs conflictedfile
